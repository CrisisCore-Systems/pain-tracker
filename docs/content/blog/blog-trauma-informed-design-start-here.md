# Trauma-Informed Design: Start Here

*A reading path through the posts that explain why Pain Tracker is built for bad days, what that means in practice, and how those constraints reach all the way down into hooks and architecture.*

---

This cluster was always present in the writing, but it did not have a front door.

So this is the front door.

If you want to understand the trauma-informed design layer of Pain Tracker, the useful sequence is not just UX advice.

It moves through four levels:

- lived experience
- product philosophy
- component and hook patterns
- system architecture

That matters because trauma-informed design is not a copywriting style in this repo.
It is a product constraint.

This is not mental-health diagnosis.
This is not a claim that software can “detect trauma.”
This is about designing software that remains usable and non-shaming when the person using it is in pain, overloaded, shaking, foggy, or trying not to fall apart.

## Why this path exists

Most discussions of trauma-informed design stay at the level of tone.

That is too shallow.

In this repo, trauma-informed design means:

- lower cognitive load under stress
- fewer shame-inducing interactions
- calmer failure states
- accessible defaults
- explicit user control during vulnerable moments

Those are interaction rules, but they are also system rules.

That is why these posts belong together.
