# Reversibility Contract

Last updated: 2026-03-20
Status: Draft for human review

Purpose: define what counts as reversible behavior in the live product, identify intentionally irreversible actions, and set release requirements for future destructive, sharing, account, and session-management features.

This document is repo-grounded. It describes current behavior when that behavior exists, and it states release requirements when the feature class is not yet live in the main app.

Core rule: reversibility is a property of storage and lifecycle, not UI wording. For every action below, the defensible claim is based on:

- what state changes
- whether prior state is retained long enough to stop or undo the mutation
- where the irreversible boundary sits
- what code or tests currently support the claim

## Scope

In scope:
- Delete, clear, reset, import, export, and privacy-sharing flows in the local-first app.
- Rollback expectations for future data-edit, account-change, permission, and device/session revocation features.
- Failure-mode behavior for crash, offline, and partial-sync conditions.

Out of scope:
- Changes to encryption, vault key handling, or export/report formats.
- Claims of cryptographic erasure.
- Archived SaaS or speculative account-management documents that are not wired into the live app.

## Scope Boundary

Live today:
- Clear all user data
- Emergency wipe on repeated unlock failure
- Scheduled report deletion
- Alert acknowledge and alert clear
- Export flows and backup export
- Backup import preview and apply
- Privacy, consent, and local usage control toggles

Release-gated or future:
- Account changes such as password, email, and phone management
- Device or session revocation beyond the current local vault lock model
- Recipient-based sharing and permission revocation beyond export-based sharing

Anything in the second list is a release requirement only until the main app ships the feature and supporting evidence.

## Definitions

- Reversible: the user can stop, undo, restore, or revoke the action without needing engineering intervention.
- Undo window: a bounded period before destructive execution where the user can cancel and prevent the mutation.
- Restore: reconstructing prior state from an explicit backup or retained local copy.
- Revoke: stopping future access or future sharing. Revoke does not imply unsending already-downloaded or already-forwarded copies.
- Rollback: returning the system to the last committed good state after a failed or abandoned change.
- Irreversible point: the point after which the app no longer offers in-product recovery for that action.

## Release-Gated Future Action Classes

These are not live in the main app. The statements below are release requirements, not implementation claims.

| Future action class | Release requirement | Evidence required before claiming support |
| --- | --- | --- |
| Account email/password/phone changes | Grace period, cancel path or recovery route, automatic rollback on failed verification, no silent lockout | Integration tests for cancel and failed-verification rollback, plus manual recovery drill |
| Permission / recipient-based sharing | Per-recipient or per-scope revoke, explicit statement of revocable vs non-revocable copies, visible change log | Integration tests for grant/revoke/failure and manual evidence showing remaining exposure on failure |
| Device / session revocation | Targeted revoke without full local wipe, pending state when offline, preserved local authority over source data | Tests for pending/offline revoke and confirmed revoke, plus manual audit of session targeting |

## What Reversible Must Mean By Action Class

### Delete and Clear Actions

- Buffered deletes must say the action is scheduled, not already final.
- Cancel or undo must prevent execution, not merely hide the message.
- When there is no post-execution restore, the UI and docs must say so plainly.
- The app must not retain hidden Class A shadow copies solely to support undo unless that design receives separate human approval.

### Export and Share Actions

- Export is reversible only before the file leaves app-managed memory.
- Closing an export flow must clear ephemeral blob URL handles and any transient in-memory references where implemented.
- Once a file is saved, copied, emailed, or uploaded outside the app, the app must not imply it can revoke those copies.
- If the future product introduces direct recipient sharing, the revoke contract must distinguish:
  - revocable remote access under app control
  - non-revocable copies already downloaded or forwarded by the recipient

### Consent and Privacy Controls

- Withdrawal must stop future collection or future sharing as soon as practical.
- Withdrawal does not retroactively unsend already-exported or already-transmitted data unless the technical path supports actual deletion and the system can verify it.
- A user-visible confirmation must say what changed, when it changed, and whether the effect is prospective only.

## Minimum Undo Windows

| Action class | Minimum requirement |
| --- | --- |
| Bulk destructive delete of Class A data | Immediate confirmation plus at least 10 seconds to cancel before execution |
| Low-risk local object delete (for example, scheduled report metadata) | At least 10 seconds undo window or equivalent non-destructive tombstone restore |
| Alert-list clear / ephemeral operational logs | At least 5 seconds undo window |
| Export and share actions | Cancel up until download or remote send begins |
| Backup import | Always cancelable before any write occurs |
| Emergency wipe / coercion-sensitive kill switch | No undo after trigger; reversibility is prospective only through prior configuration |

## Undo Window Expiry Behavior

- Clear all user data: when the 10-second window expires, `clearAllUserData()` may execute and there is no in-app restore promised after completion.
- Scheduled report deletion: when the 10-second window expires, the report is deleted from store state and the temporary undo opportunity ends.
- Alert clear: when the 6-second window expires, the temporary retained alert snapshot is discarded and restore is no longer available.
- Backup import: the cancel window ends when the user explicitly confirms import and writes begin.
- Export: the cancel window ends when the browser download or remote send begins.
- Emergency wipe: there is no post-trigger expiry phase because the action is intentionally non-reversible once invoked.

## Rollback Requirements

### Data Edits

Current state:
- Store-level `updateEntry`, `deleteEntry`, and mood-entry mutations exist.
- The live app does not currently prove a general-purpose user-facing undo or history model for entry edits.

Release requirement for any new or expanded data-edit UI:
- Single-record edits must preserve the last committed value until the new write succeeds.
- Bulk edits and imports must support preview-before-apply.
- Destructive edits must use either a time-bound undo window or a verified restore path.
- Crash during an edit must leave either the old committed state or the new committed state, never an undocumented half-state.
- If rollback depends on backup restore rather than in-flow undo, the UI must say that before commit.

### Account Changes (Password, Email, Phone)

Current state:
- Not a live feature class in the main app.

Required before release of any such feature:
- User must see old value, pending value, effective time, and recovery method.
- Sensitive credential changes must support a grace period, cancel path, or secondary recovery route.
- A failed verification step must roll back to the last working credential automatically.
- The system must avoid lockout that depends on a single device or a single channel already being changed.
- Audit trail must record that a change was requested, confirmed, canceled, or rolled back without logging the secret itself.

### Permissions and Sharing

Current state:
- The live app uses exports and privacy toggles rather than durable recipient permissions.

Required before release of recipient-based sharing:
- Grant and revoke must be explicit per recipient or per scope.
- The user must be able to see what was shared, when, and whether revocation is prospective only or actually removes access.
- Revocation failure must not be silent; the app must show the remaining exposure.
- Permission changes must produce a user-visible change log entry and a non-reconstructive audit event.

### Device / Session Revocation

Current state:
- Vault lock exists, but there is no live multi-device or remote session-revocation model.

Required before release of a session model:
- User must be able to revoke one session or device without wiping unrelated local data.
- The interface must identify the target session, last activity, and expected effect of revocation.
- Revocation must fail closed for future access but must not destroy the authoritative local data copy by default.
- If a revoke request cannot be confirmed because the device is offline, the UI must show a pending state rather than claiming success.

## Reversible Consent and Sharing Flows

### Consent Can Be Withdrawn

- Privacy and analytics toggles must be reversible prospectively.
- Withdrawing consent must stop future collection or future transmission on the local device as soon as the setting is applied.
- Where a setting is persisted locally, the withdrawn state must remain effective across reload until the user changes it again, unless the product explicitly documents a session-only control.
- The UI must state whether previously exported or previously transmitted data remains outside app control.

### Shared Data Can Be Revoked Where Possible

- For local file exports, revocation is only possible before the file leaves app-managed memory.
- For direct remote sharing, if introduced, the product must distinguish between:
  - server-side access that can be revoked
  - recipient-held copies that cannot be clawed back

### Non-Reversible By Design

- Local file exports are non-revocable after the browser hands the file off to device storage or another app.
- Consent withdrawal does not imply export revocation.
- Any future feature copy or documentation must state this boundary plainly.

### Clear User-Visible “What Changed” Logs

Current state:
- The repo contains audit and compliance logging services.
- The main user-facing app does not yet prove a complete user-visible history for all privacy, sharing, and destructive state changes.

Requirement:
- Every consent, sharing, revoke, delete, restore, or rollback action must produce:
  - immediate user-visible confirmation
  - a timestamped local change-log entry suitable for later review
  - an internal non-reconstructive audit event when the feature touches sensitive data or sharing scope

This is a release requirement for future sharing/account/session work and a current documentation gap for some existing settings flows.

## Failure-Mode Behavior

### Crash / Reload

- Buffered deletion timers must not resume after a crash or reload unless the product explicitly persists and surfaces that pending state.
- Current clear-all and scheduled-delete timers are in-memory, so the safe expectation is: crash before expiry cancels the pending action.
- Export failure must not mutate the source entries.
- Backup import must not write anything before preview and explicit `IMPORT` confirmation.
- The live backup-import path now has dedicated coverage for rollback on write failure, but crash/interruption during the write loop is still unverified.

### Offline

- Core tracking, local deletes, backup export, and local export must remain usable offline.
- Offline state must not block canceling a buffered delete because that operation is local.
- Remote submission paths, if enabled, must be optional and must not delete the local source record on network failure.

### Partial Sync

Current state:
- No general multi-device sync model is live in the main app.

Requirement for any future sync:
- Local commit remains authoritative until remote acknowledgment is complete.
- Partial sync must never tombstone or clear local data before the remote side confirms durable receipt.
- Retryable failure must preserve the user’s last good local state and expose a pending/retry state rather than a false success state.

## Current Gaps To Carry Forward Honestly

- Single-alert acknowledge is destructive and has no built-in undo.
- Clear-all data has no in-app restore after execution.
- Settings backup import has verified preview-before-write behavior and verified rollback on ordinary write failure, but crash/interruption recovery during the write loop is not yet proven.
- The main user-facing app does not yet provide a comprehensive “what changed” history for all privacy and sharing controls.
- Account-change and device/session revocation requirements are defined here as release gates for future work, not as implemented behavior.

## Verified Live Evidence Snapshot

Automated evidence currently supports these live claims:

- Clear-all data remains cancelable until the 10-second destructive timer completes.
- Scheduled report deletion is both delayed and undoable within the same 10-second window.
- Alert-log clear is undoable within 6 seconds and intentionally irreversible after that window expires.
- Settings backup import remains non-destructive until preview and explicit `IMPORT` confirmation.
- Settings backup import restores already-applied keys if a later storage write fails in the live import path.

These proofs do not extend the product contract beyond the implemented boundary. In particular, they do not imply post-commit restore for clear-all, single-alert acknowledge, emergency wipe, or crash-safe recovery from an interrupted settings-import write loop.

## Release Gate

Any change that adds or materially alters a destructive action, sharing surface, consent control, account-change flow, or session-management flow must ship with:

- updated mapping in this contract
- corresponding verification evidence from `docs/trust/reversibility-verification.md`
- human review when the change touches export boundaries, storage deletion semantics, crypto, or network sharing
