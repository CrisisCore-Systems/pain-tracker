## Section 1: The Problem
When documentation depends on memory, scattered notes, and last-minute formatting, the result is predictable: missing days, inconsistent detail, and a stressful rewrite cycle exactly when the user has the least capacity.

Common failure points:

- notes scattered across multiple places
- missing days during the hardest periods
- unclear chronology
- repeated formatting work for each request
- extra stress caused by reconstruction instead of capture

## Section 2: The Workflow Shift
The change is not magic. It is structure.

Before:

1. Symptoms tracked inconsistently.
2. Details reconstructed later from memory.
3. Timeline reformatted for each request.
4. Gaps weaken the documentation packet.

After:

1. Short structured entries captured close to the event.
2. Optional context recorded when relevant.
3. Date range selected for export.
4. Output reviewed before any sharing happens.

## Section 3: What Improved In The Composite Example
Illustrative composite metrics from the repo-side case study material:

- tracking consistency moved from irregular to much more complete
- required fields became easier to capture because the record shape was structured
- patterns and triggers became easier to explain because entries shared a common format
- packaging shifted from manual assembly to user-controlled export

Suggested supporting line:

`The improvement comes from reducing reconstruction work, not from collecting more personal data than necessary.`

## Section 4: Why This Matters
The value is not just speed. It is legibility under stress.

- a clearer timeline
- more consistent records
- less reformatting work
- user-controlled export scope
- local-first handling by default

## Section 5: Privacy Boundary
This workflow is designed around local authority.

- records stay on the device by default
- exports happen only when the user chooses to generate them
- the user reviews the output before sharing it
- the product does not promise claim outcomes or institutional acceptance
