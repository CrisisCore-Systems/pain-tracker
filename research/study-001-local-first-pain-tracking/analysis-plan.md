# Analysis Plan

## Study Purpose

This study evaluates whether a local first pain tracking tool can support routine symptom logging while increasing participant trust, reducing privacy anxiety, and avoiding unnecessary cognitive burden.

The analysis focuses on three questions:

1. Can participants complete core pain tracking tasks without assistance?
2. Does the local first privacy model increase perceived trust and control?
3. Which interface patterns create friction, confusion, or abandonment risk?

## Goals

1. Identify usability barriers in local first pain tracking.
2. Assess participant trust in data control, storage, export, and privacy.
3. Determine which features support routine logging and repeat use.
4. Identify failure points that could create stress, confusion, or perceived exposure.
5. Generate interface recommendations for a safer, more protective version of PainTracker.

## Hypotheses

### H1: Core Task Completion

Participants will be able to complete the core pain logging task within 5 minutes without moderator assistance.

### H2: Local First Trust Effect

Participants will report higher trust after receiving a clear explanation that their data is stored locally and does not require cloud account creation.

### H3: Navigation Friction

Navigation complexity will be the most common source of confusion during first use.

### H4: Export Confidence

Participants who successfully generate a report will report greater confidence in using the tool for medical, disability, legal, workplace, or advocacy contexts.

## Study Tasks

Participants will be asked to complete the following tasks:

1. Open the app and understand its purpose.
2. Create a new pain entry.
3. Edit or review a previous entry.
4. Locate the privacy or data storage explanation.
5. Generate or preview an exportable report.
6. Explain where they believe their data is stored.
7. Describe whether they would trust the tool for regular use.

## Data Sources

### Quantitative Data

1. Task completion status.
2. Time on task.
3. Number of errors.
4. Number of moderator assists.
5. Trust rating.
6. Privacy anxiety rating.
7. Cognitive load rating.
8. Likelihood of future use.
9. Export confidence rating.

### Qualitative Data

1. Participant think aloud comments.
2. Interview notes.
3. Open ended survey responses.
4. Observed hesitation points.
5. Confusion around privacy, storage, navigation, or reports.
6. Participant language around safety, control, trust, and exposure.

## Key Metrics

| Metric | Definition | Target |
|---|---|---|
| Task success rate | Percentage of participants completing each task | 80 percent or higher |
| Core logging time | Time to create a complete pain entry | Under 5 minutes |
| Moderator assists | Number of times help is required | Low or decreasing across tasks |
| Trust rating | Participant confidence in the tool | 4 out of 5 average or higher |
| Privacy clarity | Participant understanding of local storage | 80 percent correct explanation |
| Cognitive load | Participant reported mental effort | Moderate or lower |
| Export confidence | Confidence using the report externally | 4 out of 5 average or higher |

## Analysis Approach

### Quantitative Analysis

Quantitative analysis will summarize task performance and participant ratings.

Steps:

1. Clean and anonymize all records.
2. Remove direct identifiers from notes and exports.
3. Calculate completion rates for each task.
4. Calculate average, median, minimum, and maximum task times.
5. Count errors and moderator assists.
6. Summarize rating distributions using averages and frequency tables.
7. Compare results against predefined usability targets.
8. Identify tasks with the highest failure, delay, or assistance rate.

### Qualitative Analysis

Qualitative analysis will identify recurring patterns in participant feedback and observed behavior.

Feedback will be coded into themes such as:

1. Navigation confusion.
2. Privacy uncertainty.
3. Local storage trust.
4. Report or export confidence.
5. Cognitive overload.
6. Fear of data exposure.
7. Routine logging motivation.
8. Language or accessibility barriers.
9. Failure state confusion.
10. Desired features or missing context.

Each theme will be reviewed for frequency, severity, and design relevance.

## Coding Procedure

1. Review all notes and open responses.
2. Assign initial codes to recurring comments or observed behaviors.
3. Group related codes into broader themes.
4. Mark each theme as low, medium, or high severity.
5. Link each high severity theme to a specific interface issue.
6. Translate findings into design recommendations.

## Severity Ratings

| Severity | Meaning |
|---|---|
| Low | Minor friction, preference, or cosmetic issue |
| Medium | Causes hesitation, confusion, or reduced confidence |
| High | Blocks task completion, reduces trust, or creates safety concern |
| Critical | Creates perceived exposure, coercion, data loss fear, or abandonment risk |

## Protective Computing Evaluation

Findings will also be evaluated against Protective Computing principles.

| Principle | Study Question |
|---|---|
| Local authority | Did users understand they controlled their data? |
| Exposure minimization | Did users feel the app avoided unnecessary disclosure? |
| Reversible state | Did users feel able to correct, exit, or recover? |
| Explicit degradation | Did users understand what happens when something fails? |
| Consent clarity | Did users understand what was shared, stored, or exported? |
| Cognitive protection | Did the interface reduce or increase mental burden? |

## Decision Criteria

The pilot will be considered successful if:

1. At least 80 percent of participants complete the core logging task.
2. Average core logging time is under 5 minutes.
3. Most participants correctly understand the local first storage model.
4. Average trust rating is 4 out of 5 or higher.
5. No critical safety, privacy, or data loss concern is observed.
6. Findings produce clear interface improvements for the next version.

The pilot will require major redesign before expansion if:

1. Participants cannot explain where their data is stored.
2. Export or report generation creates significant confusion.
3. Navigation prevents routine logging.
4. Participants feel pressured, exposed, or uncertain about data control.
5. Multiple users require assistance for the same core task.

## Deliverables

1. Summary of key usability issues.
2. Task completion and timing summary.
3. Trust and privacy sentiment summary.
4. Cognitive load and friction summary.
5. Thematic analysis of participant feedback.
6. Protective Computing evaluation table.
7. Ranked design recommendations.
8. Limitations of the pilot.
9. Recommended changes before Study 002.

## Limitations

This pilot is exploratory. It does not establish clinical effectiveness, medical benefit, or long term adherence. It is designed to evaluate usability, trust, perceived safety, and early evidence for local first health documentation.

## Expected Output

The final report will identify whether PainTracker is understandable, trustworthy, and safe enough for vulnerable users to use as a self documentation tool. The findings will guide interface improvements and support a future larger study on adherence, privacy anxiety, and real world use.
